# V1.1 技术债与版本发布计划

## 1. 文档目的

- 记录 `V1.0` 封版后的已知非阻塞事项。
- 明确后续版本的技术债、范围边界与发布目标。
- 为 GitHub 发布、CI 持续集成和后续迭代提供统一依据。

## 2. 当前版本结论（V1.0）

- 版本定位：可运行的中文 MonoGame 贪吃蛇首个可交付版本。
- 核心层测试：`SnakeGame.Core` 单元测试已补全关键缺口，当前共 `49` 个测试用例。
- 覆盖情况：核心层行覆盖率约 `98.01%`，分支覆盖率约 `93.47%`。
- 应用层范围：`SnakeGame.App` 当前 **暂不纳入单元测试覆盖范围**。
- 应用层验证方式：以人工冒烟测试、编译验证和运行验证为主。

## 3. 已登记技术债

### TD-V1.1-001 应用层自动化测试缺失

- 类型：测试技术债
- 当前状态：已知、已接受、延期到 `V1.1`
- 影响范围：
  - `SnakeGameApp` 的菜单流转、输入接线、暂停/返回、HUD 状态呈现
  - `Program` 启动期异常日志链路
  - `LevelCatalog` 静态关卡目录装配
  - `SnakeGame.App` 中文字体资源、内容管线与运行时冒烟校验
- 当前风险：
  - 应用层回归更多依赖人工验证
  - 输入采样、界面状态切换、资源装载类问题难以被单元测试提前发现
- V1.1 处理方向：
  1. 抽离可测试的应用层协调逻辑（输入翻译、菜单状态机、运行态控制器）
  2. 评估新增 `SnakeGame.App.Tests` 或更适合的自动化冒烟测试工程
  3. 将“可自动化的非渲染逻辑”逐步从 MonoGame 直接依赖中分离

## 4. V1.1 版本目标

### 4.1 目标

- 不改变 `V1.0` 已封版的核心玩法规则。
- 优先偿还应用层自动化测试相关技术债。
- 稳定 GitHub CI 与 Tag 发布流程，形成可重复的发布节奏。

### 4.2 计划项

1. 建立应用层自动化测试方案设计文档。
2. 抽离可测试的应用层状态协调代码。
3. 为输入接线、菜单状态流转、暂停/恢复等关键场景补充自动化测试。
4. 固化 GitHub `CI + Tag Release` 发布流程并在真实仓库验证。

### 4.3 非目标

- 不在 `V1.1` 中推翻当前 `SnakeGame.Core` 结构。
- 不以重写渲染层为代价换取测试覆盖率。
- 不把所有 MonoGame 可视表现强行改写成单元测试目标。

## 5. 版本发布节奏建议

### V1.0

- 状态：封版
- 目标：保留当前实现，作为实验基线

### V1.1

- 主题：应用层自动化测试能力建设
- 入口条件：
  - `V1.0` 规则冻结
  - 核心层单元测试保持通过
  - CI 工作流可在 GitHub 上稳定执行
- 退出条件：
  - 应用层关键状态流转具备自动化回归能力
  - Tag 可一键生成 GitHub Release

## 6. 发布前检查项模板

- `dotnet build SnakeGame.sln -c Release`
- `dotnet test tests\SnakeGame.Core.Tests\SnakeGame.Core.Tests.csproj -c Release`
- Windows 环境下运行 `src\SnakeGame.App\bin\Release\net8.0-windows\SnakeGame.App.exe`
- 验证菜单、选关、暂停、返回主菜单、双击加速、进关逻辑
- 推送形如 `v1.0.0` 的 Tag，确认 GitHub Release 自动生成
